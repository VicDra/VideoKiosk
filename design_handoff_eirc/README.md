## 1. Обзор

**Продукт.** ЕИРЦ ЛО — Единый информационно-расчётный центр Ленинградской области. Базовое приложение — **CountersReader**: планшет/киоск в офисе для самостоятельной передачи показаний приборов учёта. Новый функционал — **видеосвязь с оператором** (для клиентов, которым нужна помощь) и сопутствующее **рабочее место оператора**.

**Что нужно сделать.** Внедрить дизайн-систему во все экраны приложения и реализовать поток видеосвязи. Внешний вид и поведение заданы HTML-референсами в этом бандле.

---

## 2. О файлах в бандле (важно)

Файлы в `reference-screens/` — это **дизайн-референсы, сделанные в HTML**. Это прототипы, демонстрирующие задуманный вид и поведение, **а не продакшн-код для копирования один-в-один**.

Задача — **воссоздать эти экраны в среде целевого кодекса** (React, Vue, SwiftUI, Jetpack Compose и т.п.), используя его устоявшиеся паттерны и библиотеки. Если среды ещё нет — выбрать наиболее подходящий фреймворк под планшетный/десктопный таргет и реализовать там.

`design-system/` — это переносимая основа: токены (CSS-переменные), описания компонентов, гайд. **Эти значения — источник истины.** Их нужно перенести в систему тем целевого стека (CSS-переменные / Tailwind theme / нативные ресурсы), сохранив имена и значения.

## 3. Фиделити

**High-fidelity (hifi).** Референсы — это пиксель-точные макеты с финальными цветами, типографикой, отступами и состояниями. Реализовать UI нужно **пиксель-в-пиксель**, средствами библиотек и паттернов целевого кодекса. Любое отклонение цвета/шрифта/отступа от токенов — дефект.

## 5. Взаимодействия и поведение

- **Навигация:** action-блоки на главной ведут в соответствующие потоки; «Видеосвязь с оператором» → экран #10. Кнопка «Назад/Отмена» в TopBar возвращает на предыдущий шаг.
- **Видеозвонок:** #10 (ожидание) → автоматически #11 (разговор) при ответе оператора. «Отменить»/«Завершить» → возврат на главную.
- **Оператор:** входящий вызов показывается оверлеем поверх рабочей области. «Принять» → экран разговора; «Отклонить» → следующий в очереди.
- **Анимации:** появление экрана — fade + подъём 8–12px, `--dur-base` 200ms, `--ease-out`. Пульсация колец ожидания/входящего — 1.8–2.2s бесконечно. **Все анимации обязаны уважать `@media (prefers-reduced-motion: reduce)`** (отключение).
- **Состояния кнопок:** hover — затемнение на шаг; press — лёгкий scale(0.97)/translateY(1px); focus-visible — кольцо `--focus-ring`; disabled — `--gray-200` фон, `--txt3` текст, без тени.
- **Валидация ввода показаний:** новое значение ≥ предыдущего; иначе поле-ошибка (бордер `--danger`, текст подсказки `--danger`).

## 6. Состояние (state)

- `currentScreen` — текущий экран/шаг потока.
- `account` — найденный лицевой счёт (номер, адрес, список приборов).
- `readings[]` — введённые показания по приборам (значение, дельта, валидность).
- **Видеозвонок:** `callState` = `idle | connecting | active | ended`; `elapsedSec` (таймер); `micOn`, `camOn` (булевы).
- **Оператор:** `queue[]` (источник, локация, время ожидания), `activeCallId`, `operatorStatus` = `ready | busy`.
- Данные счёта и приём показаний — запросы к биллинговому бэкенду; видеосвязь — через сигнальный сервер/WebRTC (вне дизайн-системы, но статусы UI завязаны на `callState`).

## 8. ПРОТОКОЛ ПРОВЕРКИ СООТВЕТСТВИЯ (обязательный)

После реализации Claude Code обязан прогнать проверку и зафиксировать результат по каждому пункту: **PASS / FAIL / N/A** + комментарий. Цель — **полное соответствие** реализации дизайн-системе. Любой FAIL устраняется и проверяется повторно.

### 8.A Токены (нет «магических» значений)
- [ ] **A1.** Ни один цвет в коде не задан хардкодом мимо токенов. Поиск по кодовой базе hex-литералов (`#[0-9a-fA-F]{3,6}`) и `rgb(`/`rgba(` вне файла токенов → результат пуст (кроме самих определений токенов).
- [ ] **A2.** Все `var(--*)` резолвятся. Прогнать страницу и проверить, что нет элементов с дефолтным `initial`-цветом из-за опечатки в имени переменной (см. скрипт в 8.G).
- [ ] **A3.** Размеры/отступы/радиусы берутся из шкалы (`--space-*`, `--radius-*`), а не произвольные px.
- [ ] **A4.** Тени — только `--shadow-*`. Нет инлайновых `box-shadow` с произвольными значениями.

### 8.B Типографика
- [ ] **B1.** Все заголовки (h1–h3, заголовки карточек, заголовки диалогов) — `--font-serif` (Merriweather). Нигде заголовок не отрисован Open Sans.
- [ ] **B2.** Весь текст, кнопки, подписи — `--font-sans` (Open Sans).
- [ ] **B3.** Подписи полей — ALL CAPS, 600, 12px, трекинг `--tracking-caps`.
- [ ] **B4.** Никаких сторонних шрифтов (Inter/Roboto/Arial и т.п.). Шрифты подключены (в офлайн-киоске — захостены локально, не с CDN).
- [ ] **B5.** Минимальные кегли соблюдены: текст на планшете не мельче 15px, подписи не мельче 12px.

### 8.C Цвет и роли
- [ ] **C1.** Главное действие каждого экрана (CTA) — оранжевый `--accent`. На экране не более одной первичной CTA.
- [ ] **C2.** Navy `--primary` используется только для структуры (шапки, иконки, активные значения), не для CTA.
- [ ] **C3.** Типы приборов учёта раскрашены строго по токенам (`--cold/hot/elec/gas`).
- [ ] **C4.** Статусы: успех — `--success`, ошибка/завершить — `--danger`, ожидание/внимание — `--accent`.
- [ ] **C5.** Тема только светлая; тёмная navy-сцена применяется лишь к экранам видеозвонка.

### 8.D Хром и раскладка
- [ ] **D1.** StatusBar 32px (`--primary-dk`) + TopBar 78px (`--primary`) + оранжевая accent-линия 2px присутствуют на всех экранах потока.
- [ ] **D2.** Рабочая область — фон `--bg`.
- [ ] **D3.** Скругления соответствуют роли элемента (карточки/инпуты 20, CTA-блоки 28, чипы 12, frame 14).
- [ ] **D4.** Раскладка экранов 01–12 совпадает с референсами (сетки, ширины сайдбара 320px, PiP 210×146 и т.п.).

### 8.E Доступность и тач
- [ ] **E1.** Контраст текста к фону ≥ 4.5:1 (проверить основные пары: `--txt1`/`--surface`, белый/`--primary`, белый/`--accent`).
- [ ] **E2.** Тач-цели: CTA ≥ 58px, клавиши numpad ≥ 72px, элементы десктопа оператора ≥ 44px.
- [ ] **E3.** Видимое кольцо фокуса (`--focus-ring`) на всех интерактивных элементах; навигация с клавиатуры работает.
- [ ] **E4.** Все анимации отключаются при `prefers-reduced-motion: reduce`.
- [ ] **E5.** Эмодзи не используются как функциональные иконки в проде (в референсах эмодзи — плейсхолдеры; заменить на иконочный набор, напр. Lucide outline 2px).

### 8.F Компоненты и состояния
- [ ] **F1.** Каждый компонент из `design-system/components/` имеет аналог в коде с теми же вариантами/состояниями (см. `*.d.ts` и `*.prompt.md`).
- [ ] **F2.** Состояния hover/active/focus/disabled реализованы для всех кнопок и полей.
- [ ] **F3.** Поток видеозвонка проходится целиком: главная → ожидание → разговор → завершение → главная.
- [ ] **F4.** Рабочее место оператора: входящий → принять/отклонить → очередь обновляется.

### 8.G Автоматические проверки (запустить в браузере/CI)

**Поиск нерезолвнутых переменных и хардкод-цветов** (выполнить в DevTools или через скрипт):
```js
// 1) Элементы с подозрением на нерезолвнутую переменную (цвет «съехал» в дефолт)
//    — ручная сверка: визуально сравнить с референсом.
// 2) Хардкод hex в инлайновых стилях (должно быть пусто в проде):
[...document.querySelectorAll('[style]')]
  .filter(el => /#[0-9a-f]{3,6}/i.test(el.getAttribute('style')))
  .map(el => el.getAttribute('style'));

// 3) Проверка семейств шрифтов заголовков (должен быть Merriweather):
[...document.querySelectorAll('h1,h2,h3,.card-title,.dialog-title')]
  .map(el => getComputedStyle(el).fontFamily)
  .filter(f => !/merriweather/i.test(f)); // ожидается пустой массив

// 4) Проверка тач-целей CTA (>=58px):
[...document.querySelectorAll('.cta, [data-role="primary"]')]
  .map(el => el.getBoundingClientRect().height)
  .filter(h => h < 58); // ожидается пустой массив
```
Grep по репозиторию (должно быть пусто вне токенов):
```
rg -n "#[0-9a-fA-F]{3,6}" src/ | grep -v tokens
rg -n "font-family:\s*'?(Inter|Roboto|Arial)" src/
```

## 8.5 РАЗДЕЛЬНАЯ ПРОВЕРКА ПО ПОВЕРХНОСТЯМ (две независимые итерации)

Поверхности отлаживаются **по очереди и независимо**: сначала полностью доводится и принимается **Рабочее место оператора (десктоп)**, затем — **Киоск (планшет)**. Каждый трек имеет свой скоуп экранов, свой эталон, свой прогон чек-листа 8.A–8.G (только по экранам трека) и свой критерий приёмки (gate). Киоск **не начинают**, пока трек оператора не прошёл gate.

Почему такой порядок: оператор — меньшая и более изолированная поверхность (один экран + диалоги, десктопный хром), на ней быстрее «обкатать» токены, шрифты и состояния. После её приёмки те же токены и компоненты переиспользуются на киоске с уверенностью, что основа верна.

### ТРЕК 1 — Рабочее место оператора (отлаживается ПЕРВЫМ)

- **Эталон:** `reference-screens/operator/index.html` (+ экран #12 в `app-eirc-lo.html`).
- **Скоуп экранов:** рабочее место оператора — состояния **Ожидание**, **Входящий вызов** (оверлей), **Разговор**. Очередь вызовов (сайдбар 320px), navy-шапка 64px, диалог входящего 420px.
- **Хром:** оконная рамка десктопа (не планшетный StatusBar/TopBar). Тень `--shadow-tablet`.
- **Прогнать чек-лист 8.A–8.G только по экранам оператора.** Особое внимание:
  - [ ] **OP-1.** Navy-шапка оператора 64px, аватар + имя (Merriweather) + статус «Готова принимать» (точка `--success`).
  - [ ] **OP-2.** Очередь: активный элемент — фон `--accent-soft`, бордер `#DC9830`, номер на `--primary`; остальные — бордер `--border`. Время ожидания — `--accent-dk`.
  - [ ] **OP-3.** Диалог входящего: «Принять» — зелёная `--success`, «Отклонить» — бордер `--border`; grid 1fr/1fr; «Ещё N в очереди» — `--accent-dk`. Пульсация аватара уважает `prefers-reduced-motion`.
  - [ ] **OP-4.** Экран разговора: статус «Идёт разговор» (точка `--success`) + таймер tabular-nums; панель управления (микрофон/камера стекло, завершить — `--danger`).
  - [ ] **OP-5.** Переходы состояний: Ожидание → Входящий → (Принять) Разговор → (Завершить) Ожидание; (Отклонить) → следующий в очереди.
  - [ ] **OP-6.** Заголовки оператора — Merriweather; подписи/кнопки — Open Sans; контраст белого на `--primary`/`--success`/`--danger` ≥ 4.5:1.
- **GATE 1 (критерий приёмки оператора):** все пункты 8.A–8.G по экранам оператора + OP-1…OP-6 = PASS; визуальное сравнение со скриншотом эталона совпадает; отчёт раздела 9 заполнен с пометкой «Поверхность: Оператор». **Только после этого переходить к Треку 2.**

### ТРЕК 2 — Киоск / планшет (отлаживается ВТОРЫМ)

- **Эталон:** `app-eirc-lo.html`, экраны 01–11.
- **Скоуп экранов:** поток передачи показаний (01–09) + видеосвязь со стороны клиента: ожидание (#10) и разговор (#11).
- **Хром:** планшетный — StatusBar 32px (`--primary-dk`) + TopBar 78px (`--primary`) + accent-линия 2px.
- **Прогнать чек-лист 8.A–8.G только по экранам киоска.** Особое внимание:
  - [ ] **KK-1.** Хром планшета (StatusBar 32 + TopBar 78 + accent-линия 2px) на всех экранах 01–11; рабочая область — `--bg`.
  - [ ] **KK-2.** Главная: 3 action-блока, включая «Видеосвязь с оператором» → ведёт на #10.
  - [ ] **KK-3.** Ввод показаний: numpad-клавиши ≥ 72px; тип-цвета приборов (`--cold/hot/elec/gas`); валидация «не меньше предыдущего».
  - [ ] **KK-4.** CTA каждого экрана — оранжевый `--accent`, ровно одна первичная кнопка на экран.
  - [ ] **KK-5.** Поля ввода/QR — радиусы `--radius-xl` (24); карточки/инпуты — `--radius-lg` (20); чипы — `--radius-chip` (12).
  - [ ] **KK-6.** Видеосвязь (#10/#11): navy-сцена, пульсация колец с `prefers-reduced-motion`, превью своей камеры, кнопка «Завершить» — `--danger`.
  - [ ] **KK-7.** Тач-цели CTA ≥ 58px; кольцо фокуса видно; кегли ≥ 15px (текст) / 12px (подписи).
- **GATE 2 (критерий приёмки киоска):** все пункты 8.A–8.G по экранам киоска + KK-1…KK-7 = PASS; визуальное сравнение с эталоном 01–11 совпадает; отчёт раздела 9 заполнен с пометкой «Поверхность: Киоск».

### Сводный gate
- [ ] **Кросс-проверка:** токены и компоненты, использованные в обоих треках, идентичны (один источник — `design-system/`). Никаких дублирующих/расходящихся определений цвета, шрифта, радиуса между киоском и оператором.

> Заполняется **два отдельных отчёта** (раздел 9) — по одному на трек, с полем «Поверхность: Оператор / Киоск». Сводный gate подтверждается после приёмки обоих.
