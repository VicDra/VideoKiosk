Ты — senior full-stack разработчик. Тебе нужно создать программный комплекс «Информационный киоск» для обслуживания клиентов в офисах информационно-расчётного центра (ИРЦ).

Весь комплекс работает исключительно на локальных вычислительных ресурсах (внутренняя сеть офиса). Никаких облачных сервисов, Firebase, публичных STUN/TURN-серверов. Вся инфраструктура разворачивается на собственном сервере в локальной сети.

Состав системы:
1. Клиентское Android-приложение (информационный киоск на планшете/терминале).
2. Приложение оператора (десктопное) для приёма видеозвонков.
3. Локальный сигнальный сервер (WebSocket) для установления WebRTC-соединений.
4. Локальный STUN/TURN-сервер (coturn) при необходимости — ты должен предусмотреть его конфигурацию и инструкцию по запуску, но сам код coturn мы не пишем, только конфиг.

Требования к клиентскому Android-приложению (Kotlin, минимальный API 26):
- Главный экран с тремя крупными кнопками, оформленными в стиле информационного киоска (приятный UI, крупный шрифт).
  Кнопка 1: «Информация об ИРЦ» — открывает экран с текстовой информацией об информационно-расчётном центре (адрес, часы работы, контакты). Текст статический, захардкожен в strings.xml.
  Кнопка 2: «Способы передачи показаний» — открывает экран с описанием всех возможных путей передачи показаний приборов учёта (через сайт, мобильное приложение, по телефону, через терминал и т.д.). Текст статический.
  Кнопка 3: «Видеозвонок оператору» — начинает видеозвонок.
- Экран видеозвонка: два VideoView (локальный и удалённый), кнопка «Завершить». При нажатии стартует подключение к сигнальному серверу по WebSocket (wss:// или ws://, настраиваемый IP/порт), отправляется сообщение-запрос на звонок, ждёт ответа оператора. После принятия звонка устанавливается WebRTC PeerConnection (аудио+видео). При завершении — закрытие соединения и возврат на главный экран.
- Состояния: ожидание ответа оператора (индикатор), разговор, завершение.
- Всю работу с WebRTC реализовать с помощью библиотеки Google WebRTC (org.webrtc:google-webrtc). Для WebSocket использовать OkHttp.
- Должна быть возможность настроить IP-адрес и порт сигнального сервера (можно через settings activity или конфигурационный файл).
- Архитектура: MVVM, использовать ViewModel, LiveData/StateFlow. Код должен быть чистым, с разделением на слои.

Требования к приложению оператора:
- Десктопное Java-приложение (JavaFX или Swing, предпочтительно JavaFX) или веб-приложение (HTML/JS) для приёма видеозвонков от клиентского киоска. Выбери наиболее простой и надёжный для локального развёртывания вариант. Я рекомендую десктопное JavaFX приложение, так как оно легко запускается на машине оператора без браузерных заморочек.
- Интерфейс: окно, в котором отображается статус «Ожидание вызова». Когда приходит входящий видеозвонок, появляется уведомление с возможностью «Принять»/«Отклонить». После принятия — экран видеозвонка с двумя видео (оператора и клиента) и кнопкой «Завершить».
- Оператор также подключается к тому же сигнальному серверу по WebSocket, регистрируется как оператор.
- При нескольких одновременных звонках должна быть очередь: если оператор уже разговаривает, новые входящие помещаются в очередь и обслуживаются по очереди (простой механизм на сигнальном сервере).
- Все настройки подключения к сигнальному серверу должны быть вынесены в конфигурационный файл (properties/yml).

Требования к сигнальному серверу:
- Простой WebSocket-сервер на Node.js (или Java/Kotlin, на твой выбор). Я предпочитаю Node.js (библиотека ws) для быстроты реализации.
- Протокол: JSON-сообщения.
  Типы сообщений:
  - от клиента: { type: "call", clientId: "kiosk_1" }
  - от сервера оператору: { type: "incoming_call", clientId: "kiosk_1" }
  - от оператора: { type: "accept", clientId: "kiosk_1" } или { type: "reject", clientId: "kiosk_1" }
  - далее обмен SDP offer/answer и ICE candidate между клиентом и оператором через сервер, пересылка без изменений: { type: "offer"/"answer"/"ice_candidate", target: "client"/"operator", sdp/ice: ... }
- Поддержка очереди: если оператор занят, сервер ставит входящий звонок в очередь и отправляет клиенту сообщение { type: "queued" }. Когда оператор освобождается, сервер отправляет ему следующий звонок из очереди.
- Сервер должен запускаться на локальной машине (IP в локальной сети), слушать порт (например, 8080). Никакой аутентификации, безопасность на уровне изолированной сети.

Требования к локальному медиасерверу:
- Для гарантированной работы WebRTC в сложных сетевых условиях должен быть поднят локальный STUN/TURN сервер (coturn). Ты должен предоставить готовый конфиг для coturn (turnserver.conf) с минимальными настройками для локальной сети (например, аутентификация через long-term credentials, порт 3478, список локальных IP). Инструкцию по установке и запуску включи в README.
- В коде клиентского и операторского приложений должна быть возможность указать адреса ICE-серверов (STUN и TURN), настраиваемые через конфиг. По умолчанию прописать локальный адрес (например, 192.168.1.100:3478).
